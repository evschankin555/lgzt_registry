# План создания MAX-бота для `lgzt_registry`

## Дата
2026-03-24

## Цель
Сделать бота в мессенджере MAX с тем же бизнес-функционалом, что и текущий Telegram-бот, без раздвоения данных и без деградации текущего прод-сценария.

Ключевой целевой вариант:
- `Telegram bot` и `MAX bot` работают параллельно
- используют одну и ту же PostgreSQL базу
- используют общую бизнес-логику регистрации, предприятий, статусов, SMS-подтверждения, волонтёров и админских операций
- отличаются только транспортным слоем и UI-адаптером под конкретный мессенджер

---

## Краткий вывод

### Рекомендуемое решение
Не переписывать текущего Telegram-бота и не пытаться заменить `telebot` на универсальный SDK. Вместо этого:

1. Вынести платформенно-независимую бизнес-логику из Telegram-обработчиков.
2. Добавить отдельный MAX-сервис в этом же репозитории.
3. Подключить оба сервиса к одной БД.
4. Переделать схему идентификации пользователей, волонтёров и админов так, чтобы Telegram и MAX могли существовать одновременно.

### Почему это лучший путь
- минимальный риск для рабочего Telegram-бота
- не ломает текущий production
- позволяет вводить MAX поэтапно
- сохраняет единый источник данных
- позволяет постепенно выровнять функционал обеих платформ

## Что есть сейчас в проекте

### Текущая архитектура
- Основной бот: `main.py`
- Бизнес-логика и операции с БД: `functions.py`
- Модели: `models.py`
- Подключение БД: `db.py`
- Telegram UI и состояния: `vars.py`, `modules/admin_ui.py`, `modules/user_ui.py`
- Веб-админка уже работает с той же БД: `registry_dashboard/backend/main.py`

### Ключевая проблема текущей схемы
Проект жёстко завязан на Telegram ID:
- `User.tg_id`
- `User_volunteer.tg_id`
- `User_who_blocked.tg_id`
- `admin_ids` и `superadmin_ids` в `vars.py`

В таком виде MAX-бот нельзя безопасно подключить к тем же данным без конфликтов идентичности.

---

## Ограничения и факты по MAX

По официальной документации MAX на 2026-03-24:
- в production рекомендован `Webhook`, для тестов допустим `Long Polling`
- есть inline-клавиатуры и `callback`
- есть `request_contact`
- есть редактирование сообщений
- есть ответ на callback
- есть лимит около `30 rps`
- токен передаётся через заголовок `Authorization`

Это означает, что целевой функционал Telegram-бота в MAX в целом реализуем.

Что это значит practically:
- сценарий регистрации переносим
- выбор предприятия через inline-кнопки переносим
- админ-меню переносим
- профиль пользователя переносим
- Excel-выгрузку переносим
- запрос контакта, callback и webhook доступны

## Главная архитектурная идея

### Разделить систему на 3 слоя

#### 1. Domain / Business layer
Общая логика без привязки к Telegram/MAX:
- старт регистрации
- валидация шага
- подтверждение личности по фамилии и дате рождения
- подтверждение телефона и SMS
- назначение предприятия
- завершение регистрации
- профиль пользователя
- админские операции
- работа с волонтёрами

#### 2. Platform adapter layer
Отдельные адаптеры:
- `telegram_bot/*`
- `max_bot/*`

Их задача:
- принять update от платформы
- преобразовать его в внутреннюю команду
- вызвать общий сервис
- отрисовать ответ платформенными кнопками/сообщениями

#### 3. Persistence layer
Общая БД и общие SQLAlchemy-модели/репозитории.

## Ключевое архитектурное решение по данным

### Решение A. Быстрый путь, но слабый
Добавить поля:
- `user.max_id`
- `user_volunteer.max_id`
- `user_who_blocked.max_id`

Плюсы:
- быстрее внедрить
- минимальная первая миграция

Минусы:
- схема начнёт дублироваться по платформам
- дальше будет тяжело поддерживать
- новые каналы связи добавлять неудобно

### Решение B. Правильный путь, рекомендуемый
Ввести универсальные таблицы идентичностей.

Пример:
- `user_identity`
  - `id`
  - `user_id`
  - `provider` (`telegram`, `max`)
  - `external_user_id`
  - `external_chat_id` nullable
  - `first_name`
  - `last_name`
  - `username`
  - `linked_at`
  - unique(`provider`, `external_user_id`)

- `volunteer_identity`
  - `id`
  - `volunteer_id`
  - `provider`
  - `external_user_id`
  - `display_name`
  - unique(`provider`, `external_user_id`)

- `blocked_identity_event`
  - `id`
  - `provider`
  - `external_user_id`
  - `blocked_at`
  - `linked_user_id` nullable

Плюсы:
- нормальная многоканальная модель
- Telegram и MAX живут параллельно
- проще поддерживать в будущем
- админка может видеть привязки по каналам

Минусы:
- больше объём работ на старте

### Рекомендация
Брать `Решение B`.

## Варианты реализации MAX-части

### Вариант 1. Отдельный Python-сервис на REST API MAX
Рекомендуемый вариант.

Почему:
- текущий проект уже на Python
- текущая бизнес-логика и ORM уже на Python
- не нужно вводить второй runtime ради бота
- проще переиспользовать `functions.py` и будущие сервисы

Технически:
- `aiohttp` или `httpx` для вызова MAX API
- `FastAPI` endpoint для webhook
- отдельный worker/router для MAX update handling

### Вариант 2. MAX-бот на JS/TS через официальный SDK
Возможен, но не рекомендован для этого проекта.

Минусы:
- второй язык в одном продукте
- придётся дублировать бизнес-логику либо выносить её в HTTP API
- усложняется деплой, поддержка и отладка

### Вывод
Делаем MAX-бота на Python и общаемся с MAX API напрямую.

## Риски

### Риск 1. MAX update payload отличается от ожидаемого
Особенно критично для:
- `request_contact`
- callback payload
- bot_started semantics

Снижение риска:
- сначала поднять dev-бота
- собрать реальные payload samples
- положить их в `docs/max_samples/`

### Риск 2. Telegram и MAX начнут конфликтовать за один профиль
Если оставить старую схему с `tg_id`, конфликт почти гарантирован.

Снижение риска:
- сначала identity migration

### Риск 3. Состояние диалога потеряется при рестарте
Сейчас Telegram использует memory storage.

Снижение риска:
- вынести state в БД до запуска MAX

### Риск 4. Админские роли окажутся завязаны только на Telegram
Снижение риска:
- роль хранить отдельно от платформы UI

### Риск 5. Дублирование логики между Telegram и MAX
Снижение риска:
- domain service layer до MAX adapter layer

## Критерии готовности

### Фаза миграции данных считается успешной, если
- существующие Telegram пользователи автоматически перенесены в identity-таблицы
- Telegram-бот продолжает работать без регрессий

### MAX MVP считается успешным, если
- пользователь может пройти регистрацию от старта до выбора предприятия
- в БД обновляется тот же `User`
- профиль виден в MAX
- админ в MAX видит пользователей и волонтёров

### Полный запуск считается успешным, если
- Telegram и MAX одновременно работают с одной БД
- регистрация в Telegram и MAX создаёт/обновляет одинаковые сущности
